# 需求文档

## 介绍

本文档描述一个基于 Next.js 的通用 SaaS 平台需求范围，包含**前端系统（面向终端用户）**与**后台系统（面向运营/管理员）**，并遵循**业务与皮肤分离**原则（主题/品牌配置不侵入业务逻辑）。

原型参考：`ui/` 目录中的页面（首页、登录/注册、前台控制台、后台仪表盘/订单/产品计费/皮肤管理/系统设置等）。

## 需求

### 需求 1 - 应用目录、路由与布局隔离（Admin vs Default）

**用户故事：** 作为开发者与管理员，我希望后台与前台在目录、路由与布局上严格隔离，以便清晰组织代码并降低权限与样式互相影响的风险。

#### 验收标准

1. 当请求的路由匹配 `/[locale]/admin/*` 时，系统应从 `app/[locale]/(admin)/` 目录提供页面，并与默认（前台）分组保持隔离。
2. 在渲染任意后台页面时，系统应使用 `DashboardLayout` 布局，并包含持久显示的 `Sidebar` 侧边栏。
3. 当请求的路由不匹配 `/[locale]/admin/*` 时，系统应使用前台默认布局，且不渲染后台的 `Sidebar`。

### 需求 2 - 认证与账号体系（前台/后台）

**用户故事：** 作为用户/管理员，我希望能够安全登录系统并访问对应的功能模块。

#### 验收标准

1. 在访客尚未登录时，当其访问需要登录的前台页面，系统应重定向到对应的前台登录页。
2. 在访客尚未登录时，当其访问 `/[locale]/admin/*` 下的后台页面，系统应重定向到 `/[locale]/admin/login`。
3. 当用户提交无效的登录凭据时，系统应返回带有英文错误提示 `"invalid credentials"` 的失败结果。
4. 当用户通过受支持的登录方式成功登录时，系统应创建或更新用户资料，并维持有效会话。
5. 在后台鉴权时，当存在 `ADMIN_EMAILS` 环境变量时，系统应将其视为逗号分隔的邮箱列表，去除多余空格并以不区分大小写的方式进行邮箱比对。
6. 当已登录用户访问后台页面且其邮箱不在 `ADMIN_EMAILS` 白名单中时，系统应返回 HTTP 403，并在页面展示英文文案 `"No access"`。
7. 当 `ADMIN_EMAILS` 为空或缺失时，任意用户访问后台页面，系统都应返回 HTTP 403，并在页面展示英文文案 `"No access"`。
8. 当用户邮箱在 `ADMIN_EMAILS` 白名单中时，系统应将其视为拥有完整后台管理权限，而不再细分子角色。

### 需求 3 - 前台首页与内容入口（Landing/Content）

**用户故事：** 作为访客，我希望在首页快速了解产品价值、功能、定价与内容入口，从而提升转化。

#### 验收标准

1. 当访客打开首页时，系统应展示清晰的产品价值主张、功能亮点及主要行动入口按钮。
2. 当访客点击定价入口时，系统应导航到定价/套餐模块页面。
3. 当访客点击内容/博客入口时，系统应导航到内容中心页面。

### 需求 4 - 前台控制台：套餐管理（Plans）

**用户故事：** 作为已登录用户，我希望查看可购买套餐并发起支付，以获取订阅或一次性权益。

#### 验收标准

1. 当用户已登录并访问套餐管理页面时，系统应基于服务端定价配置展示套餐列表，包括商品、币种、周期、金额及权益等信息。
2. 当用户发起下单请求时，系统应校验请求中的商品、金额、币种与周期是否与服务端定价表完全一致，否则应拒绝该请求，并返回英文错误提示 `"invalid checkout params"`。
3. 当结算会话创建成功时，系统应先写入一条状态为 `"created"` 的订单记录，再返回对应的结算标识，并引导用户跳转到选择的结算服务提供方页面。
4. 当下单请求中指定了不受支持的结算服务提供方时，系统应拒绝该请求，并返回英文错误提示 `"unsupported settlement provider"`。
5. 当成功创建结算会话时，系统应在订单中持久化 `provider_code` 与 `provider_session_id`（或等价的提供方标识），以支持后续回调对账。

### 需求 5 - 结算结果可信与回调安全（Multi-provider）

**用户故事：** 作为平台拥有者，我希望结算结果只由可信渠道驱动，并兼容多种结算方式，避免伪造回调导致权益被错误发放。

#### 验收标准

1. 在表示结算服务提供方时，系统应使用稳定的 `provider_code`（例如 `"stripe"`、`"lemonsqueezy"`、`"Alipay"`），并将其视为仅由服务端控制的枚举值。
2. 当接收到任一结算提供方的回调请求时，系统应通过统一入口 `/api/settlement-notify?provider=<provider_code>`（或等价的单一入口）接入，并根据 `provider_code` 路由到对应处理逻辑。
3. 当接收到结算回调时，系统应基于签名或密钥等方式校验回调来源合法性；校验失败时，应返回 HTTP 400，并返回英文错误提示 `"invalid notify data"`。
4. 当回调内容表示支付未成功时，系统应返回 HTTP 200，仅记录审计信息而不发放任何权益，并在需要展示给用户时使用英文错误提示 `"invalid settlement status"`。
5. 当成功回调无法通过 `order_no` 元数据或已存储的 `provider_session_id` 匹配到任何订单时，系统应返回 HTTP 404，并返回英文错误提示 `"order not found"`。
6. 当订单状态不为 `"created"` 时，再次收到对同一订单的成功回调，系统应保持订单状态不变，返回 HTTP 200 以允许提供方重试，并在审计记录中标记英文信息 `"invalid order"`。

### 需求 6 - 订单生命周期与幂等（Orders）

**用户故事：** 作为用户与运营人员，我希望订单状态准确可追溯，并避免重复回调导致重复发放权益。

#### 验收标准

1. 当订单被创建并写入数据库时，系统应生成唯一的 `order_no` 并确保该字段在数据库中唯一。
2. 在结算回调被重复或重试处理的情况下，对同一订单的多次处理结果应满足幂等性：只允许一次从 `"created"` 变更为 `"paid"`，且只发放一次对应权益。
3. 当订单完成支付且被标记为已支付时，系统应持久化 `paid_at`、`paid_email` 与 `paid_detail` 等信息，用于后续审计。
4. 当存在乱序到达的回调且后到达的回调与已支付状态冲突时，系统不得将订单状态从已支付降级，并应记录该冲突事件以便审计。

### 需求 7 - 积分/额度系统（Credits）

**用户故事：** 作为用户，我希望能够查看、获得与消耗积分/额度，用于平台的计费与激励体系。

#### 验收标准

1. 当用户查询可用积分/额度时，系统在计算结果时只应包含 `expired_at` 大于等于当前时间的记录。
2. 当已支付订单需要发放积分/额度时，系统在执行发放逻辑时应保证幂等性：同一 `order_no` 不得重复产生有效发放记录。
3. 当任一积分/额度操作失败并需要向前端返回错误时，系统应使用英文错误提示 `"credit operation failed"`。
4. 当订单已被标记为已支付而积分发放失败时，系统应保持订单处于已支付状态，记录该失败事件用于审计，并允许后续通过安全的重试机制再次尝试发放且不会重复累加积分。

### 需求 8 - 前台控制台：我的订单（My Orders）

**用户故事：** 作为已登录用户，我希望查看我的订单列表与详情，以核对支付与权益状态。

#### 验收标准

1. 当用户打开“我的订单”页面时，系统在查询订单列表时应只返回当前登录用户自己的订单记录。
2. 当用户查看某个订单详情时，系统应展示订单状态、金额、币种、周期以及（如已支付）相关支付元数据。

### 需求 9 - 前台控制台：积分管理（My Credits）

**用户故事：** 作为已登录用户，我希望查看积分余额、明细、到期时间，并了解是否为已充值/Pro 用户。

#### 验收标准

1. 当用户打开“我的积分/额度”页面时，系统在加载概要信息时应展示可用积分、到期时间线以及基于订单与积分记录推导出的充值/Pro 状态。
2. 当未登录用户调用“我的积分”相关接口时，系统应拒绝该请求，并返回英文错误提示 `"no auth, please sign-in"`。

### 需求 10 - 内容中心与文章管理（Posts/Content）

**用户故事：** 作为访客与运营人员，我希望通过内容中心发布与展示文章，支持多语言与上下线。

#### 验收标准

1. 当访客访问内容中心并请求文章列表时，系统只应返回指定语言下 `status=online` 的文章。
2. 当管理员在后台创建、更新、上线或下线文章时，系统应持久化文章状态的变化，并在前台内容中心正确体现这些变更。
3. 当文章被删除时，系统应执行软删除（或等价的非破坏性状态变更），以便保留审计与恢复的可能性。

### 需求 11 - 后台：仪表盘（Dashboard）

**用户故事：** 作为管理员，我希望在仪表盘看到核心指标（用户、订单、GMV、积分等），用于运营决策。

#### 验收标准

1. 当管理员打开后台仪表盘页面并请求统计指标时，系统应返回在所选时间范围内聚合后的核心指标数据。
2. 当统计指标暂不可用时，系统在渲染仪表盘时应优雅降级，并返回英文错误提示 `"metrics unavailable"` 而非报错崩溃。

### 需求 12 - 后台：订单管理（Admin Orders）

**用户故事：** 作为管理员/客服，我希望查询与筛选订单，处理异常订单并支持审计追踪。

#### 验收标准

1. 当管理员在后台订单管理中应用状态、时间范围、用户邮箱或订单号等筛选条件时，系统应返回经过筛选的订单列表及分页元信息。
2. 当管理员查看某个订单详情时，系统应展示完整的订单生命周期字段与支付详情，以支持审计。

### 需求 13 - 后台：产品计费管理（Products & Billing）

**用户故事：** 作为运营人员，我希望配置可售卖的产品/套餐、价格、币种与周期，并控制其在前台展示与购买。

#### 验收标准

1. 当运营人员在后台更新定价配置并保存时，系统应校验配置结构与内容，若不合法应拒绝保存并返回英文错误提示 `"invalid pricing table"`。
2. 当定价配置更新后，用户再次打开前台套餐管理页面时，系统应基于最新生效的定价配置展示套餐内容。

### 需求 14 - 皮肤/主题系统（业务与皮肤分离）

**用户故事：** 作为运营人员/管理员，我希望独立配置品牌色、圆角、Logo、暗黑模式等外观，并且不影响业务逻辑实现。

#### 验收标准

1. 当管理员更新品牌与主题配置并保存时，系统应持久化主题配置（如主色、圆角、Logo 等 token），且不得直接修改业务模块逻辑。
2. 当某个主题配置被应用时，系统在渲染任意页面时应统一从当前生效的主题配置中解析 UI token，使前台与后台外观保持一致。
3. 当主题预览模式开启时，运营人员在界面上调整任意 UI token，应能实时预览效果，但在未执行保存操作前不得写入正式配置。
4. 当主题配置内容不合法时，系统在保存时应拒绝该变更，并返回英文错误提示 `"invalid theme config"`。

### 需求 15 - 后台数据复用约束（No extra tables）

**用户故事：** 作为开发者，我希望后台复用主系统的 `models` 与 `services`，避免重复数据模型与额外建表，从而降低维护成本。

#### 验收标准

1. 在实现任意后台功能时，系统在访问数据层时应复用现有的 `models` 与 `services` 模块，而不是为后台单独定义一套数据库 Schema。
2. 在新增后台功能需要持久化存储时，系统不得仅为了后台隔离而新增专用数据表，应在现有数据模型上演进或扩展。

### 需求 16 - 系统设置与配置管理（Settings）

**用户故事：** 作为管理员，我希望在后台集中配置系统级参数（如支付、邮件、站点信息、开关），并可审计变更。

#### 验收标准

1. 当管理员在后台修改系统设置并保存时，系统应校验每个字段的合法性，非法值应被拒绝并返回英文错误提示 `"invalid settings"`。
2. 当系统设置被修改并持久化后，系统应记录修改人、修改内容与修改时间，以便审计。

### 需求 17 - 安全基线（Secrets / Least Privilege / Logging）

**用户故事：** 作为平台拥有者，我希望系统遵循安全最佳实践，避免敏感信息泄露与越权访问。

#### 验收标准

1. 在任意运行环境下，当系统访问支付相关密钥时，只能从服务端环境变量读取，且绝不能将其暴露到前端构建产物中。
2. 当未授权用户调用受保护的接口且鉴权失败时，系统应返回英文错误提示 `"forbidden"`，并避免在响应中泄露内部实现细节。
3. 当系统记录包含请求上下文的错误日志时，应对其中的密钥、令牌等敏感信息进行脱敏处理。
