# 项目说明

本文件为本项目的“宪法”（Constitution）。它定义不可违背的全局约束：技术栈边界、工程化要求、沟通规范、安全与隐私要求、一致性要求与质量基线。  
**本文件不包含任何具体技术实现细节**（例如某个 API 的具体参数、数据库字段、代码片段、版本号等），这些细节应出现在设计文档与实现代码中。

## 技术栈

- Next.js（全栈 Web 框架）
- TypeScript（默认语言）
- Node.js 生态（服务端运行时与工具链）
- 数据层：使用项目既有的数据访问层（`models/`、`services/`），避免重复建设
- 测试：单元/组件测试 + 端到端测试（覆盖前台与后台关键路径）

## 包管理规范

1. 使用稳定版依赖，**禁止**使用 beta/alpha/rc/canary 等预发布版本（除非需求文档明确豁免）。
2. 禁止在同一项目中混用多个包管理器（选择一种并坚持）。
3. 新增依赖前必须说明用途与替代方案，避免重复引入同类库。
4. 依赖升级应可回滚，并在变更记录中说明影响范围。

## 工程化规范

在编写代码时，请务必遵循以下原则：

1. **严禁重复**：任何重复逻辑必须抽象为可复用单元（函数、类、组件、模块）。
2. **检查现有代码**：实现前先确认项目内是否已有可复用能力，优先扩展而非新建平行实现。
3. **模块化设计**：功能拆分为独立、可复用、边界清晰的模块。
4. **统一接口**：使用一致的参数格式与返回值结构，错误处理策略统一。
5. **配置集中管理**：所有常量与配置项集中管理（按项目约定的目录与方式），禁止散落硬编码。
6. **优先封装**：相关功能与数据封装在职责明确的模块内，避免跨层泄漏。
7. **环境变量**：通过 `.env` 与部署环境注入配置；**禁止硬编码密钥/URL/开关**到代码中。

## 开发方式

严格遵守测试驱动开发（TDD）流程，遵循红-绿-重构循环：

- 先分析需求并拆分测试点
- 先编写失败的单元测试（Red）
- 编写最小实现使测试通过（Green）
- 在测试保护下重构代码（Refactor）

## 沟通规范

1. 所有交流使用**中文**（包括代码注释），但**UI 文案与错误提示必须使用英文**。
2. 遇到不清楚、歧义或缺失的信息应立即提问，避免带着假设推进关键决策。
3. 表达清晰、简洁、技术准确；需求与约束优先于实现偏好。
4. 仅在必要处添加注释解释关键意图、边界与权衡，避免无意义“叙述型注释”。
5. 当前任务完成后，必须给出下一步建议（例如：补设计、补测试、补文档、做回归）。

## 执行任务

1. 处理复杂问题时使用 Sequential Thinking（逐步思考）将任务拆解与收敛。
2. 涉及 UI/交互的需求与实现，必须包含端到端测试或视觉回归校验策略（例如 Playwright）。
3. 进行文件/代码检索时必须避免把 `node_modules`、构建产物与缓存目录纳入目标范围。
4. 任何自动化脚本/命令必须可重复执行、可在 CI 中运行，并避免引入环境依赖。

## 安全与隐私要求

1. **最小权限原则**：默认拒绝，按需开放访问；敏感能力必须进行鉴权与授权校验。
2. **密钥与凭据保护**：密钥仅存在于服务端环境变量与安全存储中；禁止写入仓库、日志与前端产物。
3. **支付/订单/结算安全**：结算结果必须来自可信回调并可验证真实性；所有权益发放必须具备幂等控制与审计能力。
4. **隐私最小化**：仅收集实现功能所必需的数据；对用户数据访问与导出必须可审计。
5. **日志脱敏**：日志不得包含 access token、secret、支付密钥、完整个人敏感信息等。
6. **安全默认值**：关闭不必要的调试信息与公开接口，避免信息泄露。

## 一致性要求

1. **路由与目录一致性**：后台与前台保持清晰边界与一致命名；后台入口路由与目录结构必须统一。
2. **布局一致性**：后台统一使用 DashboardLayout + Sidebar；前台使用默认布局体系，避免相互污染。
3. **错误处理一致性**：错误码/错误消息格式一致；对外错误提示为英文且不泄露内部细节。
4. **数据访问一致性**：后台复用主系统 `models/`、`services/`；不因后台隔离而单独建表或复制逻辑。
5. **主题/皮肤一致性**：皮肤系统不得侵入业务逻辑；主题 token 应全局一致并可在前后台统一生效。

## 质量基线和功能测试

- 避免编写使用 `as any`、禁用 lint 等方式绕过类型系统与检查的代码。
- 代码必须无明显 lint 错误、无编译错误，且通过最小测试集。
- 提交前检查由 Git Hooks 强制执行（如 Husky + lint-staged）。
- CI/CD 流程中集成GitHub Actions 通过多种方式实现CI/CD的自动化流程，尤其是代码审计和安全扫描。
- 测试矩阵应覆盖：单元/组件测试与端到端测试（至少覆盖关键主流程）。

## TypeScript

1. 避免使用 `any` 与 `unknown`（除非边界层并配套严格校验与封装）。
2. 避免使用不必要的 `as` 断言；优先通过类型收敛与运行时校验获得安全类型。
3. 任务完成后必须进行类型检查（例如运行 TypeScript 无输出编译检查）并确保通过。
